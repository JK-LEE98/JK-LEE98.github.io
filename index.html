API Gateway — 중앙집중 인증

각 서비스가 개별적으로 JWT를 검증하던 구조에서 Gateway가 인증을 단일 처리하도록 전환했습니다.
검증된 사용자 정보는 X-User-Id 헤더로 하위 서비스에 전달하고,
하위 서비스는 별도 인증 없이 헤더만 신뢰합니다.

  • WebFlux 기반 JWT 인증 필터 구현 (Spring Cloud Gateway)
  • Whitelist/RBAC 정책을 YAML 파일로 관리 — k3s ConfigMap으로 마운트해 이미지 재빌드 없이 정책 변경 가능
  • IpForwardingFilter(@Order(-2)) — X-Real-IP, X-Forwarded-For 헤더 위조 차단, 실제 TCP IP로 강제 덮어씌움
  • Redis Circuit Breaker + User Service fallback — Redis 장애 시에도 Gateway 인증 503 완전 제거

User Service — 토큰 탈취 대응

Troubleshooting

SSE 실시간 알림 Production 환경 문제 해결

Spring Boot / SSE / Transaction / Async / Nginx

문제 상황

프랜차이즈 주문 및 결제 관리 서비스에서 SSE 기반 실시간 알림을 운영 환경에 배포했을 때,
연결은 유지되지만 이벤트가 즉시 전달되지 않고 새로고침 이후에야 알림이 수신되는 문제가 발생했습니다.

원인 분석

로컬 환경에서는 정상 동작했기 때문에 처음에는 ConcurrentModificationException, 끊어진 emitter 관리 등을 원인으로 의심했습니다.
그러나 이를 개선해도 문제가 재현되어 코드 구조를 더 깊이 분석했습니다.

진짜 원인은 @Transactional 안에서 SSE 이벤트 전송이 동기로 실행되는 구조였습니다.

  • send() 메서드가 @Transactional로 감싸진 상태에서 DB 저장 후 바로 emitter.send() 호출
  • 트랜잭션이 열린 채로 SSE 이벤트 전송이 시도됨 — 트랜잭션 커밋 전에는 응답이 flush되지 않는 상태
  • 로컬 환경(직접 연결)에서는 문제가 드러나지 않았지만 prod 환경(nginx 프록시 경유)에서는 응답이 버퍼링되어 커밋 전까지 클라이언트에 도달하지 못함

해결 방법

  • SSE 이벤트 전송 로직을 별도 메서드 sendSseNotificationAsync()로 분리
  • @Async 적용 — 트랜잭션 커밋 후 별도 스레드에서 SSE push 실행
  • send()는 DB 저장만 담당하도록 책임 분리, SSE 전송은 트랜잭션 완료 이후에 처리
  • emitter 순회 시 복사본(HashMap copy) 사용 및 IOException 발생 시 해당 emitter 즉시 제거

결과 및 인사이트

  • Production 환경에서도 알림 이벤트 발생 즉시 클라이언트가 실시간으로 수신
  • 로컬과 prod 환경의 동작 차이를 통해 @Transactional 범위 안에서 I/O 작업(SSE, 외부 호출 등)을 동기로 처리할 경우 예상치 못한 타이밍 문제가 생길 수 있음을 확인
  • 문제를 표면(동시성 버그)에서만 보지 않고 트랜잭션 범위와 스레드 실행 흐름까지 추적해 근본 원인을 찾는 디버깅 경험

BCrypt 연산으로 인한 DB 커넥션 고갈 문제 해결

Spring Boot / Transaction / HikariCP / Monitoring

문제 상황

잡아클래스 운영 중 모니터링을 통해 로그인 요청이 집중될 때
DB 커넥션 풀이 고갈되어 다른 API 요청까지 대기 상태가 되는 현상을 발견했습니다.

원인 분석

  • login() 메서드 전체가 @Transactional로 감싸진 상태에서 BCrypt 해시 검증(수백ms)이 트랜잭션 안에서 실행되고 있었음
  • BCrypt 연산이 진행되는 동안 DB 커넥션이 점유 상태로 유지됨
  • 동시 로그인 요청이 증가하면 커넥션 풀(기본 10개)이 BCrypt 대기 요청으로 꽉 차버리는 구조

해결 방법

  • login()에 Propagation.NOT_SUPPORTED 적용 — BCrypt 연산 구간에서 트랜잭션을 보류시켜 DB 커넥션 점유 차단
  • DB write 로직을 별도 LoginWriter 빈으로 분리 — 프록시 트랜잭션으로 독립 실행되도록 구조 변경
  • self-invocation으로 트랜잭션이 적용되지 않던 내부 메서드 호출 제거 후 각 메서드에 인라인 처리

결과 및 인사이트

  • 로그인 집중 구간에서 DB 커넥션 고갈 현상 해소
  • @Transactional 범위가 외부 연산(BCrypt, 네트워크 등)을 포함할 경우 커넥션 점유 시간이 늘어날 수 있다는 것을 실전에서 확인
  • 단순히 커넥션 수를 늘리는 것이 아니라 커넥션 점유 시간 자체를 줄이는 방향으로 문제를 접근

Redis Circuit Breaker fallback 버그로 인한 간헐적 503 문제 해결

Spring Cloud Gateway / Resilience4j / Redis

문제 상황

Redis가 완전히 다운된 상태에서 인증이 필요한 API를 호출하면
Circuit Breaker가 OPEN 상태임에도 불구하고 간헐적으로 503 에러가 반환되는 현상이 발생했습니다.

특히 CB가 OPEN에서 HALF-OPEN으로 전환될 때마다 30초 주기로 503이 반복되는 패턴이 확인됐습니다.

원인 분석

  • fallback lambda에서 예외 유형을 두 가지로 다르게 처리하고 있었음
  • CB OPEN 상태(CallNotPermittedException) → RedisCircuitOpenException → User Service fallback → 정상 응답
  • Redis 실제 오류(첫 요청, HALF-OPEN probe) → RedisBlacklistException → AUTH_SERVICE_UNAVAILABLE → 503
  • 결과적으로 CB가 OPEN으로 전환되는 첫 요청과 HALF-OPEN probe 시점마다 503이 발생하는 구조

해결 방법

  • fallback lambda를 통일하여 Redis 오류 유형에 무관하게 모두 RedisCircuitOpenException으로 처리
  • Redis 실제 오류와 CB OPEN 상태를 구분할 필요가 없는 이유 — 둘 다 Redis를 신뢰할 수 없는 상황이므로 동일하게 User Service fallback으로 처리하면 됨

결과 및 인사이트

  • Redis 완전 다운 상태에서도 Gateway 인증 503 완전 제거
  • Circuit Breaker의 상태 전이(CLOSED → OPEN → HALF-OPEN) 흐름을 실전에서 디버깅하며 동작 방식을 정확히 이해
  • fallback 처리 시 예외 유형을 과도하게 세분화하면 오히려 예외 경로가 늘어나 버그를 유발할 수 있음을 확인

이메일 클라이언트 prefetch로 인한 보안 토큰 소모 문제 해결

Spring Boot / Spring Security / Redis

문제 상황

토큰 탈취 대응 기능으로 새 기기 로그인 감지 시 보안 이메일을 발송하고,
이메일 내 링크를 클릭하면 강제 로그아웃(계정 보호)이 실행되도록 구현했습니다.

그런데 실제 테스트에서 링크를 클릭하기 전에 이미 Redis의 one-time 토큰이 소모되어
링크 클릭 시 401이 반환되는 문제가 발생했습니다.

원인 분석

  • Gmail 등 일부 이메일 클라이언트는 보안·미리보기 목적으로 이메일 내 링크를 자동으로 GET 요청(prefetch)함
  • 강제 로그아웃 API가 GET 방식으로 구현되어 있었고, 이 prefetch 요청이 실제 처리를 실행해버림
  • Redis에 저장된 one-time 토큰이 사용자가 클릭하기 전에 소모되어 정작 사용자가 클릭했을 때 401 반환

해결 방법

  • GET /report-theft — 처리 없이 프론트엔드 확인 페이지로 redirect만 수행 (prefetch가 들어와도 토큰 소모 없음)
  • POST /report-theft — 사용자가 확인 페이지에서 명시적으로 버튼을 눌러야만 실제 강제 로그아웃 처리 실행
  • HTTP 메서드 의미론(GET은 안전한 조회, POST는 상태 변경)에 맞게 설계를 바로잡는 방향으로 해결

결과 및 인사이트

  • 보안 이메일 링크 → 확인 페이지 → 강제 로그아웃 흐름이 정상적으로 동작
  • GET 요청은 멱등성과 안전성이 보장되어야 한다는 HTTP 원칙을 실전 버그로 체감
  • 외부 클라이언트(이메일 앱, 크롤러 등)가 GET 링크를 자동으로 호출할 수 있다는 점을 보안 설계 시 고려해야 함

API Gateway Whitelist/RBAC 정책 관리 방식 개선

Spring Cloud Gateway / k3s ConfigMap / Flyway / Caffeine

문제 상황

초기 설계에서는 Gateway 레벨의 RBAC가 존재하지 않았고,
각 서비스가 JWT claims에서 role을 꺼내는 @CurrentUserRole을 통해 권한을 직접 검증했습니다.
이후 팀 내 논의를 통해 Gateway에서 role을 검증하고 라우팅하는 방향으로 전환하면서
Whitelist와 RBAC 정책을 Gateway 인증 필터에 하드코딩하게 되었습니다.

그런데 API가 추가되거나 접근 권한이 변경될 때마다 Gateway 코드를 수정하고
재빌드·재배포해야 하는 문제가 생겼습니다.
MSA에서 모든 요청이 경유하는 Gateway는 재배포를 최소화해야 하는 컴포넌트인데,
정책 변경이 곧 배포 이벤트를 강제하는 구조였습니다.

원인 분석

  • Whitelist와 RBAC 정책이 코드에 직접 선언되어 정책 변경과 배포가 강결합된 구조
  • JWT claims에서 role을 꺼내 판단하는 방식이어서, 사용자의 실제 역할이 변경되어도 토큰 만료 전까지 반영되지 않는 문제도 내재되어 있었음

1차 시도 — Gateway 전용 DB + Flyway + Caffeine

정책을 코드에서 분리하기 위해 Gateway 전용 DB를 두고
Whitelist와 role-path 매핑을 테이블로 관리하는 방식을 도입했습니다.

  • Whitelist 경로와 역할별 접근 가능 경로를 DB 테이블에 저장 — API 변경 시 DB 수정만으로 반영
  • Flyway 도입 — 정책 변경 이력을 마이그레이션 파일로 관리하여 환경별 일관성 유지
  • Caffeine 캐시 적용 — 매 요청마다 DB를 조회하는 성능 저하 방지
  • 한계 발견: DB 장애 발생 시 Gateway까지 장애가 전파되는 단일 장애 지점(SPOF) 문제 — Gateway 가용성이 DB 상태에 종속됨

최종 해결 — k3s ConfigMap

DB 의존성 없이 정책을 외부에서 관리할 수 있는 방법을 검토한 끝에
k3s ConfigMap을 활용한 YAML 기반 정책 관리 방식을 선택했습니다.

  • Whitelist와 RBAC 정책을 YAML 파일에 선언적으로 정의
  • dev 환경: classpath의 YAML을 직접 로드
  • prod 환경: k3s ConfigMap으로 YAML을 Pod에 볼륨 마운트
  • GitHub Actions workflow 구성 — gateway-rules.yml 변경 시 자동으로 ConfigMap을 업데이트하여 Gateway 재빌드 없이 정책 즉시 반영
  • DB 의존성 제거 — Gateway 가용성이 하위 서비스 장애에 영향받지 않는 구조

결과 및 인사이트

  • API 정책 변경 시 Gateway 재배포 제로화
  • Gateway가 외부 정책 파일 기준으로 RBAC를 처리하면서 JWT claims에서 role 제거 — 각 서비스의 @CurrentUserRole 전면 삭제, 인증·인가 책임이 Gateway로 완전히 집중됨
  • 하드코딩 → DB+Caffeine 구현 → ConfigMap 전환: 직접 구현하고 장애 전파 가능성을 발견한 후에야 더 나은 방향을 찾을 수 있었음
  • 정책(Policy)은 코드보다 변경 빈도가 높다 — 변경 빈도가 다른 관심사는 코드에서 분리해야 한다는 원칙을 실전에서 체득
